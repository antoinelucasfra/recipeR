## 3. Data Persistence — File-Based

**This app does NOT use a database.** All data is stored as RDS files under `~/.recipeR/`.

| File | Content |
|---|---|
| `~/.recipeR/recipes.rds` | Recipe list (list of lists) |
| `~/.recipeR/shopping.rds` | Shopping list (character vector) |
| `~/.recipeR/prefs.rds` | User preferences (list: `unit_system`) |
| `~/.recipeR/backups/` | Timestamped backup snapshots |

All persistence is handled exclusively through functions in `R/data_storage.R`.
Never read or write `~/.recipeR/` directly outside of those functions.

### Storage API (`R/data_storage.R`)

**Recipes:**
- `add_recipe(recipe)` — adds/updates; auto-generates `recipe_id` if missing
- `get_recipes()` — returns named list of all recipes
- `update_recipe(recipe_id, recipe)` — updates `last_modified`
- `delete_recipe(recipe_id)` — removes from list

**Ingredients (inventory):**
- `add_ingredient(ing)`, `get_ingredients()`, `update_ingredient(id, ing)`, `delete_ingredient(id)`

**Shopping list:**
- `get_shopping_list()`, `save_shopping_list(items)`

**Import/Export:**
- `export_recipes_json(path)`, `export_recipes_csv(path)` — exports current recipes
- `import_recipes_json(path, overwrite = FALSE)` — imports with dedup by `recipe_id`
- `import_recipes_csv(path)` — imports from CSV

**Backup/Restore:**
- `backup_db()` — creates timestamped backup in `~/.recipeR/backups/`
- `list_backups()`, `restore_backup(path)`

**Preferences:**
- `get_prefs()` — returns list (default: `list(unit_system = "american")`)
- `save_prefs(prefs)`

## 4. Key Utility Functions

### `ingredient_utils.R`

- `parse_fraction(str)` — parses fractions/mixed numbers to numeric (e.g. `"1 1/2"` → `1.5`)
- `parse_ingredient_line(line)` — parses raw text into `list(quantity, unit, name)`
- `unit_to_metric(amount, unit)` — converts to metric base unit; returns `list(amount, unit)`
- `metric_to_preferred(amount, type, system)` — converts metric back to preferred unit system
  - `system`: `"american"` or `"metric"`
  - `type`: `"volume"` or `"mass"`

### `density_management.R`

- `get_density(ingredient_name)` — returns density for an ingredient (g/mL); returns `NA` if unknown
- `volume_ml_to_mass_g(volume_ml, ingredient)` — converts volume to mass using density lookup

## 7. Testing

```r
# Run all tests
devtools::test()

# Run the specific test file
testthat::test_file("tests/testthat/test_ingredient_utils.R")
```

**Test coverage (4 tests in `test_ingredient_utils.R`):**
1. `parse_fraction` — handles mixed numbers (`"1 1/2"` → `1.5`), plain fractions, integers, decimals
2. `parse_ingredient_line` — extracts `quantity`, `unit`, `name` from raw text like `"1 1/2 cups flour"`
3. Unit conversion round-trip — `unit_to_metric(1, "cup")` ≈ `236.588` mL, back to `"cup"` via `metric_to_preferred`
4. Density conversion — `get_density("all purpose flour")` is not `NA`; `volume_ml_to_mass_g(236.588, "flour")` ≈ `125` ±20g

**Note:** test file uses `source('../../R/ingredient_utils.R')` directly. Do not break this
relative path without updating the test.

**Before every deploy:** run `devtools::test()`. All 4 tests must pass.

## 8. Deployment — Posit Connect Cloud

```r
# From dev/03_deploy.R — the canonical deploy command:
rsconnect::deployApp(
  appName = desc::desc_get_field("Package"),
  appTitle = desc::desc_get_field("Package"),
  appFiles = c("R/", "inst/", "data/", "NAMESPACE", "DESCRIPTION", "app.R"),
  appId = rsconnect::deployments(".")$appID,
  lint = FALSE,
  forceUpdate = TRUE
)
```

- Target: `connect.posit.cloud`, account `antoinelucasfra`
- `appFiles` must include `R/`, `inst/`, `NAMESPACE`, `DESCRIPTION`, `app.R`
- Never add `.env` files or `renv/library/` to `appFiles`
- The `.posit/` directory in `R/` contains Posit Connect deployment metadata — do not edit manually
- After deployment, verify the app runs on `connect.posit.cloud`

**Pre-deploy checklist:**
1. `devtools::test()` — all tests pass
2. `air format .` — code is formatted
3. `renv::snapshot()` — `renv.lock` is up to date and committed
4. Run locally: `recipeR::run_app()` — no startup errors
5. Deploy: `rsconnect::deployApp(...)`

## 9. Git Conventions — recipeR Specifics

Beyond root `AGENTS.md` git rules:

- No CI — every push to `main` does NOT trigger any automated checks
- You are responsible for running tests and checks manually before pushing
- Never commit `~/.recipeR/` — it is user data, not project data
- Commit `renv.lock` after every dependency change

**Commit scope examples:**
```
feat(recipe): Add recipe import from JSON
fix(units): Correct cup-to-ml conversion factor
refactor(storage): Extract backup logic to separate helpers
test(ingredient): Add tests for metric unit round-trip
chore: Bump renv.lock after adding shinyWidgets
deploy: Deploy v0.1.1 to Posit Connect Cloud
```

## 10. Hard Rules — recipeR Specifics

- Never delete `R/_disable_autoload.R` — the app will break without it
- Never edit `NAMESPACE` by hand — regenerate with `devtools::document()`
- Never access `~/.recipeR/` directly — always go through `R/data_storage.R` functions
- Never run `install.packages()` — use `renv::install()` followed by `renv::snapshot()`
- Always run `devtools::test()` before deploying — all 4 tests must pass
- Always run `air format .` before committing R code
- Never commit `.env` files, API keys, or rsconnect credentials
- Never add `renv/library/` to `appFiles` in `rsconnect::deployApp()`
- Never create `mod_*.R` files manually — use `golem::add_module("name")`
